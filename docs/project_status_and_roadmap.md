# 项目现状与扩展规划

## 1. 项目定位

`LancelotEngine` 当前是一个轻量级 2D 引擎原型仓库，核心目标不是先做完整游戏内容，而是先搭建一条清晰、可持续扩展的渲染与运行时基础链路，再逐步补齐编辑器、资源系统、项目系统和脚本能力。

从当前源码来看，项目已经从早期的 `SDL + ImGui` 编辑器原型，演进到了以 `GLFW + OpenGL + 自研 Renderer` 为核心的渲染层验证阶段。也就是说，仓库里同时存在两类内容：

- 当前已经接入主构建链路、可以作为“现阶段成果”来总结的模块
- 已经实现了较多代码，但还没有完全并入当前可执行目标的扩展模块

这两部分都很有价值，但在对外说明时应当区分“已经跑在主链路中”和“已经具备基础实现、待进一步整合”。

## 2. 当前已经实现的内容

### 2.1 可运行的应用主链路

当前程序入口已经形成完整的最小运行闭环：

1. `main.cpp` 创建 `Application`
2. `Application::Init()` 初始化窗口、图形上下文、Renderer、Shader 和相机控制器
3. `Application::Run()` 进行逐帧更新、事件轮询、场景绘制与缓冲交换
4. `Application::Shutdown()` 释放 Renderer 与窗口资源

这说明项目已经具备一个基础但完整的“应用初始化 -> 实时循环 -> 渲染输出 -> 资源释放”框架。

### 2.2 窗口系统与图形上下文

当前窗口层已经实现：

- 基于 GLFW 的窗口创建与销毁
- OpenGL 3.3 Core Profile 上下文初始化
- VSync 开关
- Framebuffer 尺寸获取与窗口最小化判断
- 窗口尺寸变化回调
- 鼠标滚轮回调
- 全屏 / 窗口化切换

这意味着项目已经具备了引擎运行所需的基本平台层能力，后续继续扩展输入、事件和多窗口支持时有比较清晰的挂载点。

### 2.3 渲染抽象层

当前渲染层不是直接把 OpenGL 调用写死在业务逻辑里，而是已经抽出了较清晰的层次：

- `GraphicsAPI`：图形后端枚举
- `GraphicsContext`：图形上下文抽象
- `RendererAPI / RenderCommand`：底层绘制命令封装
- `Renderer`：场景级提交入口
- `Renderer2D`：2D 绘制接口
- `Shader / ShaderLibrary`：Shader 编译、加载与管理
- `Material`：材质参数封装
- `Buffer / VertexArray / Texture`：GPU 资源抽象
- `platform/opengl/*`：OpenGL 平台实现

目前真正落地并跑通的是 OpenGL 后端，但整个结构已经为未来增加其他渲染后端预留了接口位置。

### 2.4 2D 渲染最小能力

当前已经完成的 2D 渲染能力包括：

- 正交相机
- 相机视图投影矩阵传递
- 四边形顶点与索引缓冲构建
- 颜色填充 Quad 绘制
- 基于 `Transform` 的平移、旋转、缩放
- 视口随窗口尺寸变化自动更新

在当前 Demo 中，场景里已经能看到一个带有以下行为的动态 Quad：

- 持续旋转
- 在 X 轴往返移动
- 按正弦节奏进行缩放脉动

这说明“时间步长驱动的对象更新 + 基础 2D 提交渲染”已经跑通。

### 2.5 相机控制系统

当前已经有一个独立的 `OrthographicCameraController`，并接入了主循环，支持：

- `W/A/S/D` 平移相机
- `Q/E` 旋转相机
- 鼠标滚轮缩放
- 窗口尺寸变化后自动重算投影范围

这部分虽然还属于原型控制器，但已经能作为后续编辑器相机、运行时相机、关卡预览相机的基础雏形。

### 2.6 基础时间系统

项目已经实现了 `Timestep`，并在主循环中使用真实帧间隔计算每帧的 `deltaTime`。这意味着：

- 动画与移动不再完全依赖固定帧数
- 后续可在此基础上继续演进为固定更新 + 可变渲染的标准游戏循环

目前这一层属于“已落地但仍然偏轻量”的状态。

## 3. 仓库中已经具备基础实现、但尚未完全并入当前主构建的内容

这部分不是空想或 TODO，而是仓库中已经存在较完整代码的能力储备。它们对项目后续演进非常重要。

### 3.1 场景数据与序列化

仓库中已经有：

- `SceneState`
- `GameObject`
- `SceneSerializer`

当前支持把场景保存为 JSON，并从 JSON 反序列化回来，字段包括：

- 对象 ID、名称
- 位置、缩放、旋转
- 纹理资源 ID / 路径
- 脚本资源 ID / 路径

这说明项目已经具备“场景资产持久化”的基础，对后续编辑器、项目系统、运行时加载都很关键。

### 3.2 资源管理系统

仓库中的 `ResourceManager` 已经实现了比较完整的纹理资源基础能力：

- 纹理路径搜索路径管理
- 路径规范化与路径解析
- 按标识符加载纹理
- 缓存复用
- 别名与真实路径映射
- 纹理释放与全量清理
- 加载失败错误信息记录

这部分说明项目已经开始从“直接读文件”过渡到“资源系统管理”，为后续编辑器资源浏览、导入和打包提供了基础。

### 3.3 资产注册表与资源导入

`AssetRegistry` 已经不是简单占位，而是做了较多事情：

- 自动识别 Texture / Audio / Text / Scene / Script 等资产类型
- 导入单个文件或整个文件夹
- 把资源复制进项目资产目录
- 生成并保存资产清单
- 通过资源 ID / 路径检索资产
- 对项目资产目录做同步扫描，发现新增和移除资源

这意味着项目已经有了“项目级资产数据库”的雏形，后续做编辑器资源面板时不必从零开始。

### 3.4 项目管理系统

`ProjectManager` 已经实现了比较明确的项目文件结构管理能力：

- 创建项目目录
- 生成 `project.json`
- 约定 `Assets / Scenes / Settings` 目录结构
- 创建不同类型的项目条目
- 删除、重命名、移动项目内资源
- 校验路径是否越界到项目目录之外

这部分非常适合作为未来“新建项目 / 打开项目 / 项目资源树”的基础。

### 3.5 原生脚本运行时

`ScriptRuntime` 的实现已经相当具体，尤其在 Windows 下：

- 支持脚本源文件动态编译为 DLL
- 支持 `OnStart / OnUpdate` 生命周期
- 支持热重编译缓存
- 能把对象的位置、旋转、缩放等上下文传入脚本
- 脚本执行后把结果写回 `GameObject`

这说明项目未来是有机会演进为“原生 C++ 脚本驱动对象行为”的，而不是只能停留在静态场景展示。

### 3.6 编辑器原型代码

仓库里仍然保留了较完整的编辑器代码结构，例如：

- `EditorUI`
- `HierarchyPanel`
- `ScenePanel`
- `InspectorPanel`
- `AssetPanel`
- `ConsolePanel`
- `EditorState`

从代码组织看，这套编辑器原型曾经已经覆盖过：

- 对象选择
- 属性检查与修改
- 资源面板浏览
- 场景操作
- 控制台占位

虽然这些文件当前没有接入 `CMakeLists.txt` 的可执行目标，但它们不是“从零开始”的状态，而是后续可以重新整合的现成资产。

## 4. 当前项目最值得强调的阶段成果

如果需要对外总结，当前最适合强调的不是“功能数量很多”，而是“底层结构已经开始成形”。可以归纳为以下几点：

- 已经跑通了一个基于 `GLFW + OpenGL` 的轻量级 2D 引擎主循环
- 已经搭建了较清晰的渲染抽象层，而不是只有临时 Demo 代码
- 已经验证了正交相机、窗口响应、2D Quad 绘制和逐帧更新逻辑
- 已经积累了资源管理、场景序列化、项目系统、脚本运行时和编辑器原型等后续扩展模块
- 仓库已经具备从“渲染验证”继续走向“编辑器 + 项目工作流 + 游戏原型”的基础

换句话说，项目当前最大的成果是“基础架构方向明确，而且很多未来功能已经有了代码基础”。

## 5. 未来可以拓展的内容

下面这些方向都和现有代码能自然衔接，属于比较现实的扩展路线。

### 5.1 完善 2D Renderer

当前 `Renderer2D` 主要验证了彩色 Quad。下一步可以扩展：

- 纹理 Quad 绘制
- Sprite 渲染
- 批处理提交，减少 Draw Call
- 排序层 / Z 层支持
- 线框、矩形、圆形、调试图元绘制
- 离屏渲染与场景预览纹理

这是把“渲染原型”升级为“可用 2D 渲染模块”的第一步。

### 5.2 引入 ECS 或更明确的组件系统

当前场景对象结构还是 `GameObject` 直存字段。后续可以演进为：

- `TransformComponent`
- `SpriteRendererComponent`
- `CameraComponent`
- `ScriptComponent`
- `ColliderComponent`
- `RigidBodyComponent`

如果进一步做 ECS，整个编辑器、序列化和运行时逻辑都会更清晰，也更适合做复杂场景。

### 5.3 完善游戏循环

现在主循环已经有 `deltaTime`，但还可以进一步升级为：

- 固定时间步长更新
- 渲染帧与逻辑帧解耦
- 暂停、单帧推进、时间缩放
- 性能统计与帧时间可视化

这一层做扎实后，物理、动画、碰撞和脚本行为都会更稳定。

### 5.4 碰撞与物理基础

对于 2D 引擎来说，最有价值的扩展之一是加入基础碰撞系统，例如：

- AABB 碰撞检测
- Tilemap 碰撞
- 地面检测、重力、跳跃
- 动态 / 静态刚体
- Trigger 与碰撞回调

这一步能直接把项目推进到“平台跳跃 Demo 可验证”的阶段。

### 5.5 重新整合编辑器

编辑器代码目前已经有原型基础，后续很适合重构接入当前 OpenGL 渲染主链路，形成：

- Hierarchy 对象树
- Inspector 属性面板
- Scene 视口
- Asset Browser
- Console / Log 面板
- Toolbar 播放控制

如果把编辑器视口和当前 `Renderer2D` 打通，项目就会从“运行时渲染 Demo”进入“可交互工具链”阶段。

### 5.6 统一资源管线

现在仓库里既有渲染层的 `Texture` 抽象，也有 `ResourceManager / AssetRegistry`。后面可以继续统一为：

- 统一的资源 ID
- 编辑器资源引用与运行时资源加载打通
- 资源元数据与缩略图缓存
- 资源丢失检测与重定位
- 构建导出时的资源收集与裁剪

这一步完成后，项目的“编辑器体验”和“运行时稳定性”都会提升很多。

### 5.7 脚本工作流升级

`ScriptRuntime` 已经提供了很好的起点，后续可以扩展：

- 脚本编译错误回显到编辑器控制台
- 脚本热重载状态提示
- 脚本字段暴露到 Inspector
- 更安全的脚本沙箱边界
- C++ 脚本之外增加 Lua / C# 等更高层脚本方案

这会显著提升项目作为“可快速验证玩法原型”的能力。

### 5.8 项目系统与导出能力

项目管理已经有雏形，未来可以补完整条工作流：

- 新建 / 打开 / 最近项目
- 项目设置面板
- 默认启动场景配置
- 一键构建运行
- 发布导出
- 模板项目初始化

这部分完成后，项目就不仅是“引擎代码仓库”，而是接近一个可用工具。

### 5.9 更完整的内容管线

后续还可以继续支持：

- Tilemap
- Sprite Sheet / Animation Clip
- 音频播放与音效管理
- 粒子系统
- 字体与文本渲染
- Prefab
- UI 系统

这些都是把项目从“课程级原型”进一步推向“更完整 2D 创作工具”的关键能力。

## 6. 建议的阶段性路线

结合当前仓库现状，比较合理的推进顺序建议如下：

### 第一阶段：把当前主链路做扎实

- 完善纹理渲染
- 加入纹理版 Sprite
- 稳定窗口、相机和时间系统
- 增加基础调试信息

目标是让当前运行程序不只是“彩色 Quad Demo”，而是具备基础 2D 场景表达能力。

### 第二阶段：整合现有储备模块

- 接入 `SceneState` 与 `SceneSerializer`
- 接入 `AssetRegistry` 与 `ResourceManager`
- 接入 `ProjectManager`
- 重新挂回编辑器面板

目标是让仓库里已有但分散的能力重新连成一条完整工作流。

### 第三阶段：推进玩法验证

- 加入脚本驱动对象行为
- 加入碰撞和平台跳跃逻辑
- 做一个小型可玩 Demo

目标是验证这套引擎结构是否足够支撑真实 2D 原型开发。

### 第四阶段：面向工具化与交付

- 完善项目管理
- 完善编辑器交互
- 完善资源导入和导出
- 做演示文档与样例工程

目标是让项目具备展示、答辩、复用和持续开发的价值。

## 7. 总结

如果用一句话概括当前项目状态，可以写成：

> 这个项目已经完成了轻量级 2D 引擎的基础运行与渲染框架搭建，并在仓库中积累了场景序列化、资源管理、项目管理、脚本运行时和编辑器原型等关键扩展模块，下一步非常适合围绕“渲染完善 -> 模块整合 -> Demo 验证 -> 工具化交付”的路径继续推进。

这是一种“底层基础已成形、上层工具正在汇合”的阶段状态。它的优势不在于现阶段功能已经全部做完，而在于后续每一块扩展都已经有明确挂点，不需要完全推倒重来。
